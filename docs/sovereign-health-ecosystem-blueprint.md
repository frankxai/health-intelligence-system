# Sovereign Health Ecosystem Blueprint

**Evidence checked:** 2026-06-22  
**Status:** architecture proposal.  
**Scope:** open ecosystem for personal health management with LLMs, agents, markdown, Obsidian, Notion, local folders, optional ChatGPT-style tools, and clinician handoff workflows. Not medical advice.

## Thesis

Health Intelligence System should not be another symptom checker. It should be a person-owned health operating system.

The system helps people collect facts, organize records, track routines, understand questions, prepare for clinicians, and keep jurisdiction-aware health documentation. It can use frontier AI models, Custom GPTs, ChatGPT Health, local LLMs, coding agents, and private infrastructure, but the person's private health record remains the source of truth.

## First Principles

1. **The person owns the operating layer.** Health records, logs, questions, and summaries live in a private vault the person controls.
2. **Clinicians own clinical decisions.** The system prepares questions, summaries, and evidence dossiers; it does not diagnose, prescribe, or choose treatment.
3. **Privacy is a mode, not a slogan.** Users choose between consumer AI, compartmentalized ChatGPT Health/Projects, regulated workspaces, private cloud, or local-only agents.
4. **Markdown is the durable working format.** Markdown works in folders, Git, Obsidian, Notion exports, plain text editors, and coding agents.
5. **Standards matter at the boundary.** Human work happens in markdown; imports/exports can map to PDF, CSV, FHIR, Patient Summary, or clinician one-pagers.
6. **Jurisdiction is a plugin.** Netherlands, Germany, Spain, Croatia, the US, China, and other systems handle records differently. The core should stay global; adapters should localize.
7. **Research is separate from personal care.** Life-science research intelligence can study diseases, literature, trials, and mechanisms; personal care artifacts stay clinician-routed.

## Privacy Modes

| Mode | For whom | Stack | Data rule |
| --- | --- | --- | --- |
| Mode 0: Public/sanitized | Anyone learning the system | Public repo, examples, docs | Fictional or sanitized data only |
| Mode 1: Consumer AI convenience | Users comfortable with consumer AI tools | ChatGPT, Custom GPTs, Projects, Notion, Obsidian | Share only what user intentionally accepts sharing |
| Mode 2: Compartmentalized health AI | Users with access to health-specific AI spaces | ChatGPT Health or dedicated health project | Keep health context separated from general chat |
| Mode 3: Regulated workspace | clinics, companies, researchers, care teams | ChatGPT for Healthcare, enterprise AI, audit logs, DPA/BAA/GDPR workflows | Follow organizational compliance policy |
| Mode 4: Local-first private | privacy-sensitive users | local folders, Obsidian, Codex/OpenCode/Hermes, Ollama/LM Studio/Open WebUI | Raw health records stay local and encrypted |
| Mode 5: Private infrastructure | technical users/organizations | self-hosted models, private APIs, encrypted object storage, access logs | Treat as security engineering, not vibes |

## Why Private GitHub Is Not The Default For Raw Health Records

Private GitHub can be useful for templates, docs, encrypted backups, and collaboration around code. It is not the safest default for raw medical records.

Risks:

- Git history preserves mistakes unless carefully rewritten.
- Private repos still depend on account security, tokens, organization access, provider policies, and legal process.
- PDFs and images can contain hidden metadata.
- It is easy to accidentally push a private file from a working directory.
- Family history, diagnosis timelines, insurance records, and portal exports are sensitive even without obvious identifiers.

Recommended default:

1. Public GitHub: core templates and fictional examples.
2. Local encrypted vault: raw records and private logs.
3. Optional private GitHub: sanitized markdown, encrypted archives, or template customization only.
4. Clinician exports: reviewed PDFs or printed summaries.

## Data Flow

```text
portal / clinician / wearable / user notes
  -> raw private records
  -> record index
  -> daily and weekly logs
  -> LLM-assisted summary draft
  -> human review
  -> clinician-facing export
  -> post-visit update
```

The LLM can help at every arrow except the clinical decision itself.

## Core Workflows

### 1. Doctor Visit Prep

- gather reason for visit;
- build symptom or issue timeline in the user's own words;
- list medications, supplements, allergies, and constraints;
- identify top three questions;
- list records to bring;
- confirm follow-up owner, results timeline, urgent-contact criteria, and next appointment.

### 2. Disease Navigation Brief

Use when a person has a known diagnosis or suspected condition and wants to understand the process.

Allowed:

- organize diagnosis as written by clinician;
- summarize public education sources;
- create a glossary;
- identify records needed;
- prepare questions for clinician;
- track treatment options offered by care team.

Not allowed:

- infer diagnosis;
- interpret labs or imaging;
- rank treatments;
- recommend medication, supplements, fasting, or delay;
- reassure or alarm beyond clinician-stated facts.

### 3. Nutrition And Fitness Loop

Allowed:

- track meals, hydration, appetite, training, soreness, sleep, recovery, energy, and constraints;
- summarize patterns;
- prepare questions for clinician, dietitian, physical therapist, trainer, or coach;
- plan ordinary wellness experiments within user-stated goals.

Not allowed:

- prescribe medical nutrition therapy;
- recommend supplements for disease treatment;
- design rehab for injury;
- override clinician restrictions.

### 4. Health Record Index

- index labs, visit notes, imaging, pathology, medications, referrals, discharge letters, insurance documents, and portal messages;
- distinguish record location from record content;
- mark missing records and requests;
- keep raw files outside public repo.

### 5. Jurisdiction Adapter

- country;
- residence status;
- insurance type;
- care portals;
- right-to-access route;
- cross-border care route;
- language needs;
- emergency and urgent-care pathways;
- data-sharing cautions.

## ChatGPT / Custom GPT Pattern

Users who accept cloud AI can create a dedicated health project or Custom GPT with these rules:

- "You are a health records organizer and doctor-visit preparation assistant."
- "Do not diagnose, interpret results, prescribe, or choose treatment."
- "Ask for source dates and distinguish public education from personal medical advice."
- "When I provide records, summarize facts and questions for my clinician."
- "Never store unnecessary sensitive details in memory."
- "Prefer one-page clinician summaries, timelines, record checklists, and question lists."

When ChatGPT Health is available, use it as the compartmentalized health space rather than mixing health files into ordinary chats.

## Local-First Pattern

For privacy-sensitive users:

- Obsidian vault or plain folder for markdown;
- encrypted local disk;
- encrypted backup;
- Codex/OpenCode/Hermes to scaffold, search, summarize, and update files;
- Ollama or LM Studio for local model inference;
- Open WebUI if a local ChatGPT-style interface is desired;
- no public network exposure for local LLM services;
- no raw records in public Git.

## Notion Pattern

Notion can work well for dashboards, habits, appointments, and non-sensitive operating views. For raw records or highly sensitive details, prefer local encrypted storage unless the user knowingly accepts Notion's cloud data model and account/security risks.

## Obsidian Pattern

Obsidian is a strong default because it is local markdown, human-readable, agent-readable, and portable. Use folders for records and templates, tags for topics, and links for appointments, records, and questions.

## Setup Path For Codex

Codex should guide a user through:

1. choosing a privacy mode;
2. creating the private vault;
3. copying templates;
4. setting up an Obsidian vault or plain folder;
5. creating a record index;
6. creating the first doctor-visit prep file;
7. creating the first weekly nutrition/fitness loop;
8. optionally configuring ChatGPT Project, Custom GPT, local LLM, or Open WebUI;
9. verifying no raw health data is in public git;
10. creating the first clinician export.

## Success Metrics

- User can find any major record in under two minutes.
- User can prepare a one-page doctor-visit brief in under fifteen minutes.
- Weekly review produces a short pattern summary and question list.
- No raw health data enters public repositories.
- Every source-backed artifact has an evidence-check date.
- Every clinician-facing artifact is reviewed by the person before sharing.
